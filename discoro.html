asyncoro

asyncoro is a Python framework for developing concurrent, distributed programs with asynchronous completions and coroutines. See API for Coroutines for more details.

API for Distributed Coroutines

By default, asyncoro does not start network services so that only coroutines running in that asyncoro instance can communicate; see API for Coroutines for details. However, when network services are enabled, messages can also be exchanged among coroutines running on multiple asyncoro instances over network. Following additional API describe network features of asyncoro for distributed programming.

AsynCoro(node=None, udp_port=None, tcp_port=0, ext_ip_addr=None, name=None, secret='', certfile=None, keyfile=None, notifier=None) enables network services when either node or udp_port options is given. node is either host name or IP address and udp_port is port number where asyncoro starts network services. If udp_port is set to 0, then port 51350 is used. Thus, initializing AsynCoro with AsynCoro(udp_port=0) is simplest way to start asyncoro for distributed programming.

Other options to AsynCoro that may be useful in networked operations are:

  • By default, any available port is used for tcp_port where asyncoros exchange messages. However, this port can be set to a specific number, which can be useful when dealing with firewall/port forwarding.
  • name, if given, must be a string that is unique for each instance of asyncoro that communicate. If this option is not given, it is set to string representing location where network services are running. Using name, the location where a specific asyncoro instance is running can be found with the method locate_peer(name). This location can then be used to find and use services, such as monitoring coroutines, or start remotely callable coroutines.
  • secret is a string that must be set to same for all asyncoros that communicate. It is used to compute a hash that is used and checked in all communication, so that only asyncoros that use same secret can be peers. Note that the messages are not encrypted; if security of messages is needed, SSL options certfile and/or keyfile can be used.
  • certfile and keyfile are as per SSL module.
  • ext_ip_addr must be IP address of gateway/firewall when asyncoro is used behind that gateway/firewall and other peers are outside that gateway/firewall. With this option, peers can communicate with asyncoro running on a local network and other peers are on remote network.

When AsynCoro is initialized with network services for distributed programming, following methods are available to find and use services at peers.

  • The (singleton) AsynCoro instance, which can be obtained with AsynCoro.instance(), has following additional methods:
    • locate_peer(name, timeout=None) finds and returns location where peer AsynCoro with name is running. This location can be used in other methods. The timeout is the maximum number (or fractions) of seconds for finding location. If timeout seconds elapse (if no peer with name is running or reachable), None is returned. This method must be used with yield.
    • locate_coro(name, location=None, timeout=None) finds and returns reference to coroutine registered with name. If location is given (e.g., obtained by locate_peer), only the peer running at that location is queried; otherwise, all peers are queried. If coro is successfully located, the return value is an instance of _RemoteCoro (see below). This method must be used with yield.
    • locate_channel(name, location=None, timeout=None) finds and returns a reference (an instance of _RemoteChannel) to channel registered with name. If location is given (e.g., obtained by locate_peer), only the peer running at that location is queried; otherwise, all peers are queried. The reference returned can be used to send or deliver messages, just as if it is an AsyncChannel (although AsyncChannel is at remote peer). This method must be used with yield.
    • register_RCI(method) registers generator method method as remotely callable. RCI stands for Remote Coro Invocation (or Remotely Callable Interface). Once registered, coroutines at peers can locate and invoke the method (i.e., run the method with coroutine where it is registered).
    • locate_RCI(name, location=None, timeout=None) finds and returns location where remotely callable method with name has been registered. This location can then be used for running (remote) coroutine with that method. This method must be used with yield.
    • run_RCI(location, name, *args, **kwargs) requests peer asyncoro at location to run generator method name (as a coroutine) with arguments *args and keyword arguments **kwargs. If coroutine is started successfully, the return value is a reference to that coroutine (i.e., instance of _RemoteCoro). This can be used to monitor and send messages. If security (in addition to SSL and/or 'secret' features) is needed so that coroutines can be remotely started with only appropriate credentials, the caller can be expected to execute loc = yield locate_RCI('peer_method') rcoro = yield run_RCI(loc, 'peer_method') try: yield coro.monitor(rcoro) rcoro.send(creds) # ... except MonitorException as exc: # process exception and generator method 'peer_method' execute creds = yield coro.receive() if not valid_creds(creds): raise Exception('invalid credentials') # ...
    • peer(node, udp_port=51350) locates peer asyncoro at given node and udp_port. If all asyncoros are running on a local network, they can find each other automatically through UDP broadcasting. However, if an asyncoro is running on a remote network, it needs to be explicitly located with this peer method. It is not needed to run this method on both sides - if one asyncoro successfully runs this method, they will find each other. This method must be used with yield.
  • AsyncChannel's register(name=None) registers it with AsynCoro so that coroutines at peers can locate and obtain a reference with locate_channel. If name is None, name of the channel initialized with is used for registering. The name must be unique, at least at the asyncoro where it is registered. If there are duplicate names at different asyncoros, finding a channel with locate_channel should be used with specific location (peer). Note that instances of SyncChannel cannot be registered, as it has no reference method.
  • Coroutine's register(name=None) registers it with AsynCoro so that coroutines at peers can locate and obtain a reference with locate_coro. If name is None, the name of generator method used (i.e., coro.name)is used. The name must be unique, at least at the asyncoro where it is running. If there are duplicate names at different asyncoros, finding a coroutine with locate_coro should be used with a specific location (peer).
  • locate_coro and run_RCI return an instance of _RemoteCoro on success. This instance can be used to monitor and send messages, as if it is running locally, as shown in snippet for run_RCI.
